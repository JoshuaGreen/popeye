Hello out there !

Here you see the `new' popeye-sources. Some remarks on them:

We - this is Elmar Bartel and Norbert Geissler -  got the
original source from Thomas Brand who got it from Ph. Schnoebelen.
This original source from Schnoebelen was a monolithic PASCAL program
for an exotic PASCAL Compiler. It was neither standard nor TURBO.
Thomas Brand did some minor errorcorrecting on this PASCAL source.
We have a lot of machines (big one and smaller ones, UNIX MX500 to IBM-XT)
but none of them with a pascal compiler which could compile this source.
Therefore we decided to convert this source to C.
I (Elmar Bartel) believed that it would be a inferior task, when we could
automatically translate the code from PASCAL to C. I saw a similiar
job done, when converting TeX-Pascal output from tangle to C. So
I startet using web2c (this is the program which does the job for the special
output of tangle) to convert the POPEYE code to C, with the intention
put it through and compile the C code. But it wasn't as easy, because
web2c cannot handle nested procedure declarations. They are missing
in C and tangle produces not such code, so there was no neccessarity
to implement this in web2c. But we had to convert the pascal source
to fit to the constraints of web2c. Finally after a lot of editing, we
got the source through the converter.
After a few further corrections by Norbert this code compiled successfully 
on a UNIX machine. Small additional corrections and the compiler
produced a runnable popeye which was functionally equivalent to its
MS-DOS original.
But this C-Code was not clean, for reading or modifying.
Also it seemed to us runtime inefficient. So I manually
edited the whole C source again, rewrote some procedures for efficiency,
splitted the source in six smaller parts, made it ANSI conform - and
the result of this, is what you can look at.
Popeyes generated from this code executes up to 30% faster than the first
C Version. The popeye generated with TURBO C 2.0 (Small Memory Model)
is 15% faster than the original version from Schnoebelen.

This C-Source should compile under any UNIX System. It also compiles under
DOS with TURBO C 2.0 from Borland. I.e. it is ANSI C conform.
See the Makefile for further instructions.

It compiles with TURBO C on an ATARI, and with MPW on macintosh computers.

If you are doing some additional work on Popeye, let us know what you
plan to do. May be we or someone else did it, and you needn't do it again.
Please help us to keep a consistent version of popeye.

The routines ErrorMsg and Message need now only one parameter,
which is the number. If this message needs a second parameter,
then it is logged with one of three routines:
	logStrArg       /* StringArgument */
	logIntArg       /* NumberArgument */
	logChrArg       /* CharacterArgument */
This has the advantage of not pushing two arguments on the stack
when most times only one is needed. And, more important we need not
any longer the stdarg, to keep the ANSI-compilers happy.
							04.02.1993  ElB

Then I added a second method to handle the language dependend strings:
There ist the old method, of keeping all the strings in seperate
files. This is needed on some architectures where every byte of
memory is valuable (f.i. 64kB SmallMemoryModel DOS).
The second method is to compile in all messages. This is done if
you define MSG_IN_MEM in the makefile.
If this is defined, another version of genpystr is compiled which
generates a suitable c-source which is included in pymsg.c
To achieve coherent handling of the languages there are two litte
files
	pylang.h
	pylang.c
Look at them and you will see. They are just too simple to
describe here.
This was done to satify the needs of Harald, who had problems
whith the old mechanism on his mainframe implementation of C.
I hope this solution is generic enough to produce valid
